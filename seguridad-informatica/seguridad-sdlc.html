La seguridad no es un complemento que se añade al final del desarrollo; es una propiedad que debe tejerse en cada fase del SDLC. Los ataques más costosos de la historia no explotaron fallos de cifrado sofisticados, sino omisiones en el análisis de requerimientos o decisiones descuidadas en el diseño. Este blog integra las herramientas fundamentales para construir software seguro desde su origen: el modelo de amenazas STRIDE, los requerimientos y el diseño seguros, los árboles de ataque y las técnicas de testing especializadas.

I

STRIDE: modelo de clasificación de amenazas

STRIDE es un modelo creado por Microsoft en 1999 (Loren Kohnfelder y Praerit Garg) que clasifica las amenazas de seguridad en seis categorías. Cada letra representa un tipo de ataque y, en el análisis de amenazas, se mapea contra las propiedades de seguridad que viola.

S
Spoofing · Suplantación

El atacante finge ser otro usuario, proceso o componente del sistema para obtener acceso no autorizado.

Ej: robo de sesión, phishing de credenciales, ARP spoofing. Contramedida: autenticación fuerte (MFA, certificados).
T
Tampering · Manipulación

Modificación no autorizada de datos en tránsito o en reposo, alterando la integridad del sistema.

Ej: inyección SQL, modificación de parámetros HTTP, MITM. Contramedida: firmas digitales, hashing, HTTPS/TLS.
R
Repudiation · Repudio

Un actor niega haber realizado una acción que sí ejecutó; sin auditoría, el sistema no puede probarlo.

Ej: borrado de logs, transacciones anónimas no trazables. Contramedida: auditoría firmada, registros inmutables.
I
Information Disclosure · Exposición

Acceso a información que el actor no debería ver: datos personales, secretos de negocio o configuraciones internas.

Ej: errores verbose, directory traversal, fugas en logs. Contramedida: cifrado, control de acceso, error handling genérico.
D
Denial of Service · Denegación

El atacante agota recursos (CPU, memoria, red) para impedir que usuarios legítimos accedan al servicio.

Ej: flood de peticiones HTTP, ataques a la capa de aplicación. Contramedida: rate limiting, CDN, escalado automático.
E
Elevation of Privilege · Escalada

Un usuario con privilegios bajos obtiene capacidades de un rol superior, comprometiendo el control de acceso.

Ej: vulnerabilidades en sudo, IDOR, JWT sin firma verificada. Contramedida: principio de mínimo privilegio, RBAC estricto.

STRIDE no es una lista de vulnerabilidades: es una lente de análisis. Se aplica a cada componente del diagrama de flujo de datos (DFD) — procesos, almacenes, flujos y entidades externas — para generar una lista exhaustiva de amenazas antes de escribir una sola línea de código.

II

Seguridad en el análisis de requerimientos

Integrar la seguridad en la fase de requerimientos significa capturar explícitamente las necesidades de protección del sistema al mismo tiempo que los requerimientos funcionales. Las técnicas más efectivas son los abuse cases (casos de abuso) y los misuse cases, que modelan el comportamiento de actores maliciosos.

Tríada CIA como base de requerimientos no funcionales

🔒

Confidencialidad

Solo los actores autorizados deben acceder a la información. Se traduce en requerimientos de control de acceso, cifrado en reposo y en tránsito, y gestión de identidades.

✅

Integridad

La información solo puede modificarse de formas autorizadas y conocidas. Implica auditoría, validación de entradas, firmas digitales y controles de integridad de bases de datos.

🌐

Disponibilidad

El sistema debe ser accesible cuando los usuarios lo necesiten. Se captura en SLA de tiempo de actividad, planes de recuperación ante desastres y protección contra DoS.

Abuse Cases y Misuse Cases

Los abuse cases son una extensión de los casos de uso que modelan explícitamente cómo un actor malicioso puede explotar el sistema. A diferencia de un caso de uso estándar que describe el flujo normal, un abuse case describe el flujo de ataque.

Elemento Caso de uso normal Abuse case / Misuse case
ActorUsuario legítimoAtacante / actor malicioso
ObjetivoLograr una función del negocioDañar, robar o interrumpir el sistema
ResultadoValor para el usuarioDaño al sistema o a sus datos
UtilidadGuía el desarrollo funcionalGuía controles de seguridad
HerramientaCasos de uso UML, User StoriesDiagrama de amenazas, STRIDE

En la práctica, por cada caso de uso que se levanta, el equipo de seguridad debe preguntarse: «¿Cómo podría un atacante abusar de esta funcionalidad?». Las respuestas se convierten en requerimientos de seguridad trazables que acompañan al caso de uso hasta las pruebas de aceptación.

III

Seguridad en el diseño del sistema

La fase de diseño es donde las amenazas identificadas con STRIDE se convierten en decisiones arquitectónicas concretas. Un diseño seguro no restringe las funcionalidades; define las barreras que hacen que violarlas sea computacionalmente costoso o imposible.

01

Mínimo privilegio (Least Privilege)

Cada componente, proceso o usuario opera con los permisos mínimos necesarios para cumplir su función. Limita el radio de explosión si un componente es comprometido.

02

Defensa en profundidad (Defense in Depth)

Múltiples capas de controles de seguridad independientes: firewall → WAF → autenticación → autorización → auditoría. Un atacante que rompe una capa enfrenta la siguiente.

03

Superficie de ataque mínima

Reducir el número de puntos de entrada expuestos: deshabilitar servicios innecesarios, cerrar puertos, ocultar versiones de software y limitar los endpoints públicos.

04

Fallo seguro (Fail Secure)

Si un componente falla o lanza una excepción no esperada, el sistema debe denegar el acceso por defecto, no otorgarlo. Los errores nunca deben revelar información sensible.

05

Separación de privilegios y responsabilidades

Descomponer operaciones críticas de modo que ningún componente individual pueda comprometer el sistema completo. Patrón aplicado en sistemas bancarios y certificados digitales.

06

No confiar en la entrada del cliente (Never Trust Input)

Toda entrada proveniente del exterior — parámetros HTTP, archivos subidos, datos de APIs — debe validarse, sanitizarse y codificarse antes de ser procesada o almacenada.

IV

Árboles de ataque: modelado formal de amenazas

Los árboles de ataque, introducidos por Bruce Schneier en 1999, son estructuras jerárquicas que representan la lógica de un ataque. El nodo raíz es el objetivo del atacante; los nodos hijos son las formas de alcanzarlo. Los nodos intermedios pueden combinarse con operadores OR (basta uno) o AND (se necesitan todos).

Ejemplo: «Obtener acceso administrativo a la aplicación»

Obtener acceso administrativo
  • OR Comprometer credenciales de administrador
    • OR Ataque de fuerza bruta
      • ANDSin límite de intentos
      • ANDContraseña débil en uso
    • Phishing / ingeniería social
    • Filtración de base de datos de usuarios
  • OR Escalada de privilegios en la aplicación
    • Explotación de IDOR en endpoint de admin
    • JWT sin verificación de firma
    • Inyección de rol en parámetro oculto
  • OR Comprometer el servidor subyacente
    • ANDRCE mediante dependencia vulnerable
    • ANDProceso con privilegios de sistema

Los árboles de ataque permiten cuantificar el riesgo asignando costo, probabilidad y factibilidad a cada hoja. El camino de menor costo para el atacante revela la prioridad de las contramedidas a implementar. Son especialmente útiles durante el diseño para comparar architecturas bajo el criterio: «¿qué tan difícil es llegar a la raíz del árbol?».

V

Técnicas de testing de seguridad

Las pruebas de seguridad no son una etapa al final del ciclo; se distribuyen en todo el SDLC. Cada técnica tiene un momento óptimo y un tipo de defecto que encuentra mejor que las demás.

Análisis estático

SAST

Analiza el código fuente sin ejecutarlo. Detecta vulnerabilidades como inyecciones, desbordamientos y manejo inseguro de datos directamente en el repositorio.

Fase: Codificación / CI pipeline
Análisis dinámico

DAST

Prueba la aplicación en ejecución enviando entradas maliciosas a través de sus interfaces. Simula ataques reales sin acceso al código fuente.

Fase: Testing / Staging
Análisis interactivo

IAST

Agente instrumentado que monitorea el comportamiento de la aplicación durante las pruebas funcionales, correlacionando tráfico con ejecución interna para detectar fallos en tiempo real.

Fase: QA / Pruebas funcionales
Prueba de intrusión

Penetration Testing

Simulación controlada de un ataque real por un equipo especializado (red team). Combina técnicas manuales y automatizadas para encontrar rutas de compromiso completas.

Fase: Pre-producción / Revisión periódica
Inyección aleatoria

Fuzzing

Envía entradas inesperadas, malformadas o aleatorias a la aplicación para provocar crashs, desbordamientos o comportamientos anómalos no anticipados en el diseño.

Fase: Codificación / Testing
Revisión manual

Secure Code Review

Inspección por pares orientada a seguridad con checklist derivado de OWASP Top 10 y CWE/SANS 25. Detecta lógica de negocio insegura que las herramientas automatizadas pasan por alto.

Fase: Pull requests / Diseño detallado

Conclusión

Integrar seguridad en el SDLC no es una práctica opcional para sistemas críticos: es el estándar mínimo de responsabilidad profesional en el desarrollo de software. STRIDE proporciona el vocabulario para nombrar y clasificar amenazas; los requerimientos de seguridad garantizan que esas amenazas tengan dueño y criterio de aceptación; el diseño seguro traduce la teoría en arquitectura; los árboles de ataque cuantifican el riesgo; y las técnicas de testing cierran el ciclo verificando que los controles implementados resisten los ataques para los que fueron diseñados.

Un equipo que recorre este flujo completo construye software que no solo funciona, sino que resiste.
