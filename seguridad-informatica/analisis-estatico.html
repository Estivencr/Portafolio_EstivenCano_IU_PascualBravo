El análisis estático examina el código fuente sin ejecutarlo — como un revisor experto que lee un manuscrito antes de imprimirlo. Es la técnica más temprana del ciclo de seguridad: actúa directamente en el editor del desarrollador o en el pipeline de CI antes de que el programa corra por primera vez. Su ventaja es su velocidad y alcance; su desafío, distinguir los hallazgos reales de los falsos positivos. Este blog explora cómo funciona, qué técnicas emplea, qué herramientas lo implementan y cómo integrarlo efectivamente en DevSecOps.

I

¿Qué es el análisis estático?

El análisis estático de código (Static Application Security Testing, SAST) es el proceso de examinar el código fuente, bytecode o binario de una aplicación sin ejecutarla, con el objetivo de detectar vulnerabilidades de seguridad, defectos de calidad y violaciones de políticas de codificación.

A diferencia del análisis dinámico (DAST), que necesita que la aplicación esté en ejecución para encontrar fallos, el SAST trabaja directamente sobre los artefactos de código: archivos fuente, librerías, configuraciones e infraestructura como código (IaC). Esto lo convierte en la primera línea de defensa y la más barata de operar, porque los defectos se detectan antes de compilar o desplegar.

II

¿Cómo funciona?

El proceso interno de una herramienta SAST sigue una cadena de pasos que transforma el código fuente en una lista priorizada de hallazgos de seguridad.

1

Parseo y construcción del AST

La herramienta analiza la sintaxis del código y construye un Abstract Syntax Tree (AST) — una representación estructurada del programa en forma de árbol que preserva la semántica sin importar el formato del archivo.

2

Análisis de flujo de datos (taint analysis)

Se rastrean las variables desde su origen (source) — entrada del usuario, parámetros HTTP, archivos — hasta los puntos donde son consumidas (sink) — consultas SQL, comandos del sistema, salidas HTML. Si un dato no sanitizado llega a un sink peligroso, se genera un hallazgo.

3

Análisis de flujo de control

Se construye un grafo de flujo de control (CFG) que representa todas las rutas posibles de ejecución. Permite detectar condiciones de carrera, código inalcanzable, y lógica de autenticación defectuosa.

4

Coincidencia de patrones y reglas

El motor compara el código contra un catálogo de patrones conocidos de vulnerabilidades (CWE, OWASP Top 10). Por ejemplo: uso de eval() con datos externos, funciones criptográficas deprecadas o sentencias SQL concatenadas.

5

Priorización y reporte

Los hallazgos se clasifican por severidad (crítica, alta, media, baja) y se generan reportes con la ubicación exacta en el código, la regla violada, el nivel de confianza y la remediación sugerida.

III

Técnicas de análisis estático

Flujo de datos

Taint Analysis

Sigue el recorrido de datos no confiables desde la entrada hasta los puntos de uso crítico. Es la técnica más efectiva para detectar inyecciones SQL, XSS y command injection.

Detecta: SQL injection, XSS, path traversal, SSRF.
Flujo de control

Control Flow Graph (CFG)

Modela todas las rutas de ejecución posibles. Permite detectar condiciones de autenticación que pueden ser saltadas, recursos que no se liberan en todos los caminos o lógica de autorización inconsistente.

Detecta: broken access control, resource leaks, dead code.
Semántica

Abstract Interpretation

Ejecuta el programa de forma abstracta sobre todos los estados posibles simultáneamente, sin ejecutarlo realmente. Permite probar ausencia de ciertos errores con garantía matemática.

Detecta: desbordamientos de entero, accesos nulos, buffer overflows.
Patrones

Pattern Matching

Compara el AST o el código fuente contra una biblioteca de patrones de vulnerabilidades conocidas. Rápido de implementar y fácil de extender con reglas personalizadas.

Detecta: uso de APIs inseguras, algoritmos criptográficos débiles, hardcoded secrets.
IV

Herramientas SAST principales

El ecosistema de herramientas de análisis estático es amplio. Algunas son de uso general; otras están especializadas por lenguaje o tipo de proyecto.

SonarQube
Java · JS · Python · C# · 30+ lenguajes

Plataforma de inspección continua de calidad y seguridad. Se integra nativamente en pipelines CI/CD (Jenkins, GitHub Actions, GitLab CI). Dashboard centralizado con tendencias históricas.

Community Edition: gratuita · Enterprise: de pago
Semgrep
Multi-lenguaje · Reglas personalizables

Motor de análisis basado en patrones con sintaxis de reglas simple y expresiva. El repositorio público de reglas cubre OWASP Top 10, CWE y patrones de frameworks específicos.

OSS: gratuito · Pro/Teams: de pago
Bandit
Python

Herramienta especializada de la OpenStack Security Group. Analiza el AST de Python buscando llamadas inseguras, uso de módulos peligrosos y malas prácticas de cifrado.

Open source · Apache 2.0
CodeQL
C/C++ · Java · JS · Python · Go · Ruby

Motor de análisis de GitHub que trata el código como una base de datos consultable. Permite escribir consultas personalizadas en QL para detectar patrones complejos de vulnerabilidad.

Gratuito para repos públicos · GitHub Advanced Security para privados
Checkmarx SAST
Enterprise · 35+ lenguajes

Solución empresarial líder del mercado. Ofrece análisis de flujo de datos incremental, integración con IDEs y gestión centralizada de hallazgos a escala organizacional.

Comercial · Licencia por proyecto
ESLint + plugins
JavaScript · TypeScript

Linter extensible con plugins de seguridad (eslint-plugin-security, no-unsanitized) que detecta vulnerabilidades comunes en el ecosistema JS directamente en el editor.

Open source · MIT
V

Fortalezas y limitaciones

El análisis estático es poderoso pero no infalible. Entender sus límites permite combinarlo correctamente con otras técnicas de seguridad.

Fortalezas
  • Detecta vulnerabilidades antes de ejecutar el código
  • Cubre el 100 % del código, incluidas rutas poco probadas
  • Totalmente automatizable en CI/CD
  • Señala la línea exacta del defecto
  • Sin impacto en ambientes de producción
  • Escala fácilmente a codebases grandes
Limitaciones
  • Tasa de falsos positivos que consume tiempo del equipo
  • No detecta vulnerabilidades de tiempo de ejecución ni de configuración
  • Lógica de negocio insegura es difícil de modelar con reglas
  • Requiere ajuste de reglas por proyecto para ser preciso
  • No reemplaza las pruebas de penetración manuales

El SAST no encuentra todos los bugs; encuentra los bugs que pueden ser descritos formalmente. Es exactamente por eso que debe combinarse con DAST, revisión de código manual y penetration testing — ninguna técnica por sí sola basta.

— Gary McGraw, Software Security: Building Security In, 2006

Conclusión

El análisis estático es la primera y más accesible barrera de seguridad en el ciclo de desarrollo. Su capacidad de operar directamente en el código fuente, antes de cualquier compilación o despliegue, lo convierte en el aliado natural del principio shift-left de DevSecOps. Integrado en el pipeline CI/CD, transforma la seguridad de una revisión periódica en una verificación continua que acompaña cada commit.

Su complemento necesario son el análisis dinámico, la revisión manual y las pruebas de penetración: juntos cubren los ángulos que ninguna técnica puede cubrir sola.
